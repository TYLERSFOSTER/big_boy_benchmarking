## Purpose

The purpose of this stage is to turn the full standard gauntlet into a readable,
auditable, and reusable evaluation surface.

It has two jobs:

1. Generate human-readable evaluation readouts from the suite's artifact tables.
2. Preserve durable system-learning notes when an evaluation exposes confusion,
   design gaps, protocol issues, or integration problems.

The generated evaluation readout should answer:

- What did we run?
- Which stage passed, warned, or blocked?
- What did the suite conclude?
- Which claims are supported?
- Which claims are not supported?
- What should be run or designed next?
- Where are the raw artifacts?

The system-learning archive should answer:

- What did this evaluation teach us about the benchmark system?
- What confused the Project Owner, evaluator, or future reader?
- Which protocol or implementation change should follow?
- Which issue should be preserved even if the generated readout is regenerated?

## Prior Work This Blueprint Uses

This blueprint directly reuses the documentation workflow that emerged during
counterpoint evaluation work:

- Raw run artifacts live inside the repository under `docs/evaluations/...`.
- Generated human-readable reports live next to the evaluation, not in `/tmp`.
- Readouts are generated by applying an explicit protocol file to an explicit
  readout source file.
- The correct prompt shape is:

  `execute docs/prime_directive/artifact_table_to_readable_document_protocol.md at <readout_source.json>`

- Generated readouts should include badges for rapid status recognition.
- Generated readouts should include turn-taking space for evaluator/consultant
  clarification.
- Conversation that becomes durable design memory should be copied or distilled
  into `docs/design/system_learning_from_evaluations/...`, rather than relying
  on generated README sections as permanent memory.
- The protocol must not fabricate Project Owner turns or attribute answers to
  the Project Owner unless the Project Owner actually supplied them.

This blueprint also inherits the documentation lesson from the counterpoint
one-third and noisy-rate diagnostic sequence: confusing or negative-looking
results are often system-learning results, not merely failed evaluations.

## Relationship To Prime Directive Protocols

This stage must follow the repository prime-directive docs, especially:

- artifact tables must be translated into readable documents through the
  explicit artifact-table readout protocol
- generated readouts must not invent human attribution
- generated readouts must preserve or create appropriate turn-taking space
- generated readouts must be grounded in table/manifests, not memory
- durable lessons learned belong in design-side system-learning folders

If the readout protocol and this blueprint disagree, the protocol should be
updated before implementation proceeds.

## README Requirements

The top-level generated README must be brief enough to scan and detailed enough
to orient a new reader.

Recommended sections:

1. Title
2. Badge block
3. One-paragraph result
4. What this suite tested
5. Current conclusion
6. Stage status table
7. Key numbers
8. Claim boundary
9. Links to detailed readouts
10. Artifact provenance
11. Clarifying conversation

### Badge Block

The badge block should appear at the top of the README.

Recommended badges:

- suite status
- artifacts complete
- structural readiness
- schema candidates
- training health
- target calibrated
- paired comparison status
- claim status
- liftability semantics
- provenance repo artifacts

The badge block is for visual recognition. It must not replace prose
interpretation.

### One-Paragraph Result

The first paragraph should say, plainly:

- whether the suite completed
- whether it found usable candidates
- whether it trained them
- whether it calibrated a target
- whether it ran a paired comparison
- what claim, if any, is supported

Example form:

`This gauntlet run completed structural diagnostics, found <N> usable schema
candidates, trained <N> tower candidates, calibrated <target>, and produced
<claim-status> for <candidate> against <baseline>.`

If the suite blocks before comparison, the paragraph should say where and why.

### Stage Status Table

The stage status table should contain:

- stage number
- stage name
- status
- key output
- downstream consequence
- detailed link

Allowed status values:

- `complete`
- `complete_with_warning`
- `blocked`
- `not_run`
- `legacy_only`

### Current Conclusion

The conclusion must be bounded.

Allowed conclusion forms:

- structural readiness conclusion
- candidate discovery conclusion
- training health conclusion
- threshold calibration conclusion
- paired comparison conclusion
- blocked/inconclusive conclusion

The README must not claim broad tower superiority from a first gauntlet result.

### Claim Boundary

The README must explicitly state:

- fixed environment instance
- fixed artifact run label
- fixed budget
- fixed target policy
- fixed candidate selection policy
- whether the result is smoke, development, or serious
- whether the comparison is claim-bearing

### Clarifying Conversation

The README should include a section for turn-by-turn clarification.

Recommended heading:

`## Clarifying Conversation`

Recommended initial structure, expressed here as placeholders rather than
transcript headings:

```
Clarification slot 1, evaluator-authored content:

_Add clarifying question or interpretation here._

Clarification slot 1, consultant-authored response:

_Add response here._
```

On subsequent regeneration, the protocol should preserve existing populated turn
pairs and add an empty pair only if no turn pair exists.

The generated readout must not fabricate Project Owner or Evaluator content.

- `pass`
- `warning`
- `blocked`
- `negative`
- `positive`
- `inconclusive`
- `not_applicable`

### Badge Color Policy

- green: pass, complete, positive signal
- yellow/orange: warning, limited, inconclusive

## System-Learning Archive Contract

Generated readouts are current-result surfaces. They are allowed to be
regenerated.

Durable design memory belongs in:

`docs/design/system_learning_from_evaluations/plate_support_standard_gauntlet_v001/`

Recommended structure:

- `README.md`
- `01_readout_conversation_archive.md`
- `02_issue_and_correction_notes.md`
- `03_follow_up_design_questions.md`

Additional subfolders may be created for specific issues.

### Archive Trigger Conditions

Create or update the system-learning archive when:

- the Project Owner identifies a conceptual confusion in the generated readout
- an evaluation result reveals a possible implementation bug
- an evaluation result reveals a protocol gap
- a negative or surprising result needs design interpretation
- an LLM or engineer made a wrong assumption that should be prevented later
- a generated readout conversation becomes important enough to preserve

### Attribution Policy

The archive must preserve attribution.

Allowed labels:

- Project Owner
- Evaluator
- Consultant
- Engineer
- Codex

Do not put words in the Project Owner's mouth.

If a note is an inference by Codex/Consultant, label it as such.

### Archive Contents

## Human-Readable Interpretive Rules

### If The Suite Stops At Stage 1

The readout should say the environment is not yet structurally ready for the
gauntlet, then name the blocked surface.

### If The Suite Stops At Stage 2

The readout should say no usable contraction/schema candidate was found, then
summarize whether the failure was collapse, construction, or executability.

### If The Suite Stops At Stage 3

The readout should say candidate discovery failed to select a candidate, then
show the blocked candidate classes.

### If The Suite Stops At Stage 4

The readout should say selected candidates were not trainable under the current
runtime/budget, then distinguish learning failure from runtime/liftability
failure.

### If The Suite Stops At Stage 5

The readout should say no calibrated target was feasible, then explain whether
the issue was budget, target sparsity, unstable returns, or insufficient signal.

### If The Suite Reaches Stage 6

The readout should state the paired comparison result with bounded claim
language.

### If The Result Is Positive

Use "limited positive signal" unless the design explicitly authorized stronger
language and the budget supports it.

### If The Result Is Negative

Do not write "the tower does not work" unless the evaluation specifically
supports that broad claim. Prefer:

`This run did not produce a positive signal for this candidate under this budget
and target.`

### If The Result Is Inconclusive

Say why:

- target too easy
- target too hard
- budget too small
- pair units incomplete
- candidate runtime unstable
- baseline/candidate difference too small
- direct and no-contraction baselines disagree

## Implementation Notes

### Reuse Existing Counterpoint Readout Machinery

The counterpoint generated readouts provide the closest pattern for:

- badge generation
- method/result/runbook pages
- artifact indexes
- clarification turns
- repo-local artifact roots
- system-learning archives

But the PlateSupport gauntlet readout should be suite-level, not just a single
evaluation readout. It must show stage progression.

### Avoid Hidden "Last Run" Semantics

The readout source must explicitly point to the artifact root.

Do not implement a command whose semantics depend on whichever run happened
last. That was a prior source of confusion.

### Do Not Flatten The Stage Story

The readout should not jump straight to Stage 6 if earlier stages contain useful
diagnostic information.

For PlateSupport, the diagnostic path is part of the result:

- structural task difficulty
- schema/tower availability
- candidate discovery
- training health
- target calibration
- paired comparison

Readers should be able to see where the evidence came from.

## Open Questions For Project Owner

These are consultant-authored open questions, not Project Owner statements.

### Question 1: Single README Or Per-Stage READMEs

Should the first implementation generate one top-level README with linked
per-stage result files, or should each stage also get its own README?

Consultant recommendation:

My recommendation is one top-level README plus linked per-stage result files for
the first implementation. Per-stage READMEs can come later if the suite readout
becomes too large.

### Question 2: System-Learning Archive By Default

Should every completed PlateSupport gauntlet run create a system-learning folder,
or should the archive only be created when a human/LLM identifies a durable
lesson?

Consultant recommendation:

My recommendation is to create the archive only when there is a durable lesson.
The generated readout can include `results/system_learning_prompt.md` every time,
but an archive folder should mean "there is something worth preserving."

### Question 3: Badge Density

How many badges should appear at the top of the suite README before it becomes
too noisy?

Consultant recommendation:

My recommendation is 8 to 10 badges maximum at the top: suite status, artifacts,
structural readiness, candidates, training health, target calibration, paired
comparison, claim status, liftability semantics, and provenance. More detailed
status can live in `results/stage_status.md`.

### Question 4: Clarifying Conversation Labels

Should generated PlateSupport readouts label clarification entries by role, for
example evaluator-authored clarification and consultant-authored response, or by
specific participant names only when those participants actually supplied the
content?

Consultant recommendation:

My recommendation is to use role labels in generated evaluation readouts unless
the Project Owner is explicitly writing the content. This avoids accidental
false attribution while still giving us a conversational surface for
clarification.
